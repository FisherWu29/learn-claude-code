# Agent 和 Skill 的区别

这份文档整理了本次讨论里关于 **Agent**、**Skill**、**Subagent**、**Team** 的区别，重点回答一个常见困惑：

> 为什么自定义 Agent 看起来很像 Skill？

结论先说：

- **Agent = 谁来做事**
- **Skill = 做事时查什么说明书**

---

## 1. 为什么容易混淆

Agent 和 Skill 表面上都可能包含这些内容：

- 角色说明
- 行为规范
- 工作流程
- 工具使用建议
- 输出格式要求
- 注意事项

所以它们看起来都像一段 prompt、一个配置文件、或者一份说明文档。这也是最容易让人困惑的地方。

但它们真正解决的问题不一样。

---

## 2. 一句话区分

### Agent
定义的是：

- 你是谁
- 你负责什么
- 你默认怎么思考
- 你的工作边界是什么

也就是：**身份 / 岗位 / 角色壳子**。

### Skill
定义的是：

- 某类任务怎么做
- 某个领域的知识和最佳实践是什么
- 某个工作流的步骤和检查项是什么

也就是：**知识 / SOP / 操作手册 / 专业插件**。

---

## 3. 最直观的类比

### Agent = 员工 / 岗位
例如：

- 测试工程师
- 架构师
- 代码审查员
- 前端工程师

### Skill = 员工手册 / 说明书 / 检查清单
例如：

- 代码审查清单
- PDF 处理说明
- MCP 开发指南
- 发布前检查清单

所以：

- **Agent 是“人”**
- **Skill 是“这个人工作时会翻阅的资料”**

---

## 4. 从“身份”还是“知识”来判断

如果你不确定一段内容该放到 Agent 还是 Skill，可以问自己一句：

> 这段内容是在描述“身份”，还是在描述“知识”？

### 更适合写进 Agent 的内容

如果它描述的是一个角色长期稳定的工作方式，通常适合写进 Agent：

- 角色职责
- 默认目标
- 决策风格
- 沟通方式
- 工具偏好
- 行为边界

例如：

- 架构师先给方案，再讨论实现，不急着直接改代码
- reviewer 默认找风险，不负责直接做大改
- tester 先复现问题，再补测试，再跑验证

这些属于“这个角色一贯如此”。

### 更适合写进 Skill 的内容

如果它描述的是某类任务的专门知识或步骤，通常适合写进 Skill：

- 某个技术领域的知识说明
- 某项工作的 SOP
- 某个流程的 checklist
- 某种文件或工具的处理方法

例如：

- PDF 文本提取方法
- MCP server 开发流程
- 安全审查 checklist
- Git 提交规范

这些属于“谁需要谁加载”。

---

## 5. 一个简化公式

### Agent

```text
role + behavior + responsibility
```

### Skill

```text
knowledge + procedure + checklist
```

---

## 6. 它们之间的关系

Agent 和 Skill 不是二选一，而是常常配合使用。

一个 Agent 在工作时，可能会按需加载一个或多个 Skill。

例如：

### tester Agent
职责：

- 设计测试
- 跑测试
- 分析失败原因
- 汇报结论

它工作时可能会使用：

- `code-review` skill
- `pytest-workflow` skill
- `release-checklist` skill

### architect Agent
职责：

- 澄清边界
- 列方案
- 做 tradeoff 分析

它工作时可能会使用：

- `mcp-builder` skill
- `security-review` skill
- `database-design` skill

所以更准确的理解是：

> **Agent 是角色壳子，Skill 是知识插件。**

---

## 7. 自定义 Agent、Skill、Subagent、Team 的关系

这几个概念经常一起出现，也容易互相混淆。可以这样理解：

### 自定义 Agent
回答的是：

> 以什么身份工作？

它定义：

- 角色
- 职责
- 风格
- 边界

### Skill
回答的是：

> 工作时依据什么知识和流程？

它定义：

- 专业知识
- 工作步骤
- 最佳实践
- 检查项

### Subagent
回答的是：

> 这次任务怎么派出去做？

它强调：

- 一次性委派
- 独立上下文
- 做完返回
- 生命周期短

### Team
回答的是：

> 多个代理怎么持续协作？

它强调：

- 多角色
- 持续存在
- 消息通信
- 生命周期管理

所以：

- **Agent / Skill** 更像“定义层”
- **Subagent / Team** 更像“协作模式层”

---

## 8. 自定义 Agent 会用在 Subagent 还是 Team？

答案通常是：**两边都可以用。**

### 在 Subagent 场景中
可以派出一个带明确角色的子代理，例如：

- `reviewer` subagent
- `tester` subagent
- `architect` subagent

这时自定义 Agent 提供的是：

- 更稳定的角色行为
- 更一致的任务处理方式

### 在 Team 场景中
也可以让团队成员各自带有稳定角色，例如：

- alice = developer
- bob = tester
- charlie = architect

这时自定义 Agent 提供的是：

- 长期岗位分工
- 清晰角色边界
- 更稳定的协作风格

因此最准确的说法是：

> **Agent 定义的是身份，不绑定某一种协作机制。**

---

## 9. Subagent 和 Team 在上下文上的区别

这也是本次讨论里一个关键点。

### Subagent
通常：

- 不继承主会话完整上下文
- 主要拿到主 Agent 派发给它的任务 prompt
- 有自己的独立上下文
- 做完后返回结果或摘要

更像：**一次性外包工**。

### Team Teammate
通常也：

- 不共享主会话完整上下文
- 创建时拿到 lead 派给它的 prompt
- 后续依靠自己的历史 + inbox 消息继续工作
- 有自己的独立上下文和生命周期

更像：**正式团队成员**。

所以 Team 并不是“所有成员共享同一个大上下文”，而是：

> **多个独立上下文，通过消息机制协作。**

---

## 10. 为什么会觉得“自己没充分利用自定义 Agent”

这是很常见的感受。

因为很多时候，Claude Code 自己会：

- 自动生成临时 subagent
- 临时创建几个 team 角色
- 用完就销毁

这其实没问题，而且对很多任务已经足够有效。

真正值得优化的，通常不是“强行多开 team”，而是：

> **把你高频重复使用的角色沉淀成稳定的 Agent 模板。**

例如：

- reviewer
- tester
- architect

如果一个角色你会反复用到，并且希望它：

- 风格稳定
- 输出一致
- 约束明确

那就值得做成自定义 Agent。

---

## 11. 什么时候做成 Agent，什么时候做成 Skill？

### 更适合做成 Agent
当它是一个你经常复用的岗位或角色：

- reviewer
- tester
- architect
- frontend-dev
- backend-dev

### 更适合做成 Skill
当它是一份领域说明书或操作指南：

- React 性能优化指南
- Python 测试规范
- Git 工作流
- MCP 开发手册
- 安全审查 checklist

---

## 12. 最容易踩的两个坑

### 坑 1：把所有内容都塞进 Agent
这样 Agent 会越来越胖：

- 既有身份
- 又有领域知识
- 又有 checklist
- 又有具体 SOP

结果会变得难维护，也不利于复用。

### 坑 2：把角色定义写成 Skill
例如 Skill 里写：

- 你是一个测试工程师
- 你是一个架构师

这其实已经在定义身份，不太像 Skill，而更像 Agent 了。

---

## 13. 最实用的最终理解

如果只记一句，就记这句：

> **Agent 是角色壳子，Skill 是知识插件。**

再展开一点：

- **Agent = 谁来做**
- **Skill = 怎么做 / 参考什么做**
- **Subagent = 一次性怎么派出去做**
- **Team = 多个代理怎么持续协作做**

---

## 14. 面向实际使用的建议

如果你想更好地利用 Claude Code 里的这些能力，可以采用一个简单策略：

### 主 Agent 直接做
适合：

- 小任务
- 不需要分工的任务

### 默认使用 Subagent
适合：

- 独立子问题
- 想隔离上下文
- 做完就回的任务

### 使用“带角色定义的 Subagent”
适合：

- 高频重复任务
- 你希望输出风格稳定

### 使用 Team
适合：

- 任务时间长
- 明显需要多角色协作
- 需要多轮消息沟通和持续分工

### 沉淀少量高频 Agent
建议优先沉淀 2~3 个：

- reviewer
- tester
- architect

这样最容易见效，也不会过度设计。

---

## 15. 总结

这次讨论最核心的几个结论是：

1. **Agent 和 Skill 看起来像，但解决的问题不同**
2. **Agent 定义身份，Skill 定义知识**
3. **Subagent 和 Team 都可以使用自定义 Agent**
4. **Team 不是共享同一份上下文，而是多个独立上下文通过消息协作**
5. **真正值得沉淀的是你高频重复使用的角色，而不是把所有内容都做成自定义 Agent**

如果只记最短版本：

> **Agent = 谁来做事**  
> **Skill = 做事时查什么说明书**
