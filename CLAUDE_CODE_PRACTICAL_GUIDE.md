# Claude Code 实战使用建议清单

这份文档不再按教程章节讲，而是直接回答一个更实际的问题：

> 学完这些 agent 思路后，今天在真实使用 Claude Code 时，哪些值得立刻用？哪些应该按需上？

目标是减少“学了很多概念，但不知道什么时候该用”的感觉。

---

## 1. 一个最实用的总原则

先记一句：

> **默认简单，只有在复杂度真的出现时，再逐层升级。**

也就是说：

- 小事别一上来就开 team
- 能主 agent 解决的，不要硬拆
- 需要隔离时再开 subagent
- 真有长期分工再上 team
- 真有并行代码修改风险再上 worktree

---

## 2. 日常工作时的决策顺序

建议按这个顺序判断：

### 第 1 层：主 Agent 直接做
适合：
- 小任务
- 一次性问题
- 只需读少量文件
- 不需要分工

典型例子：
- 改一个小 bug
- 看一个报错
- 改个文档
- 解释一段代码

---

### 第 2 层：用 Subagent
适合：
- 局部独立子问题
- 想隔离上下文污染
- 让主 agent 保持清爽
- 做完就回

典型例子：
- “去找这个 bug 的根因”
- “帮我审这个模块的风险”
- “跑一轮测试并汇总失败原因”
- “先生成一个方案草稿回来”

**实战建议**：
如果你发现主 agent 为了一个局部问题要读太多边角文件，那往往就是 subagent 的信号。

---

### 第 3 层：给 Subagent 套角色定义
适合：
- 你经常重复同一类委派
- 你希望输出风格稳定
- 你总要反复说同样的话

典型角色：
- reviewer
- tester
- architect

**实战建议**：
如果某个角色你用到 3 次以上，就值得沉淀成自定义 Agent。

---

### 第 4 层：上 Team
适合：
- 任务持续时间长
- 需要多角色持续协作
- 需要消息来回沟通
- 角色边界清晰

典型分工：
- developer：实现
- tester：验证
- architect：审方案
- reviewer：风险把关

**实战建议**：
不要为了“看起来高级”就开 team。只有在“持续分工”真的能减少你的主上下文负担时，team 才值得。

---

### 第 5 层：上 Worktree
适合：
- 多个 agent 真要并行改代码
- 风险改动想在隔离目录里做
- 想保留多个独立实验分支
- 害怕互相污染未提交改动

**实战建议**：
只要开始出现“两个代理都可能修改同一个仓库工作区”的情况，就应该认真考虑 worktree。

---

## 3. 什么最值得优先自定义

如果你不想维护太多 Agent，最推荐先做 3 个：

### 1）reviewer
职责：
- 找 bug
- 找边界问题
- 找安全/性能/可维护性风险
- 用结构化格式输出 review

为什么值得：
- 高频
- 可复用
- 风格稳定收益很大

---

### 2）tester
职责：
- 设计测试点
- 补测试
- 跑测试
- 汇总失败原因

为什么值得：
- 非常高频
- 流程容易标准化
- 和开发角色边界天然不同

---

### 3）architect
职责：
- 先拆问题
- 列方案
- 做 tradeoff
- 不直接扎进实现细节

为什么值得：
- 大改动前很有价值
- 能显著减少“直接开写导致返工”

---

## 4. Skill 最适合沉淀什么

Skill 更适合沉淀的是：

- 某技术栈 SOP
- 调试 checklist
- 发布 checklist
- 代码审查清单
- 某类工具的使用说明
- 某领域的排障手册

例如：
- React 性能优化指南
- Python 测试规范
- Git 提交流程
- MCP server 开发说明
- 安全审查 checklist

**不要把“你是 tester / architect”这种身份定义写进 skill。**
那更像 Agent，而不是 Skill。

---

## 5. 什么时候后台任务真的有价值

后台任务适合：
- 安装依赖
- 跑测试
- 跑构建
- 拉取大批日志
- 任何“慢但不必立即阻塞当前思考”的动作

但你要牢记：

> **后台化只优化等待期间能做什么，不优化关键路径本身。**

所以：
- `npm install` 在跑，你可以去写配置文件
- `pytest` 在跑，你可以去读代码或整理测试点

但如果：
- 下一步必须等测试结果决定

那最终还是得等。

---

## 6. Team 最常见的误区

### 误区 1：觉得 team 就是共享同一上下文
不是。

正确理解：
- 每个 teammate 都有自己的上下文
- 它们通过消息同步信息

### 误区 2：为了“更高级”而强行开 team
这通常只会增加管理成本。

### 误区 3：team 适合所有任务
不是。team 更适合：
- 大任务
- 持续多轮协作
- 明确角色边界

### 误区 4：team 一定比 subagent 好
不是。很多时候 subagent 更轻、更快、更省脑子。

---

## 7. 一个简单的实战选择器

### 情况 A：一个小 bug
用：**主 agent**

### 情况 B：一个 bug 涉及多文件排查
用：**subagent**

### 情况 C：要先出方案再动手
用：**architect subagent**

### 情况 D：写完后想独立审查风险
用：**reviewer subagent**

### 情况 E：写完后想系统补测试
用：**tester subagent**

### 情况 F：一个较大功能，需要实现、测试、设计协作
用：**team**

### 情况 G：多个代理要并行改代码
用：**team + worktree**

---

## 8. 面向真实仓库的最小升级路线

如果你想在真实工作里逐步吸收这些能力，不建议一次性全上。推荐路线：

### 第一步
先学会：
- 主 agent + subagent
- 少量高频自定义 Agent

这是收益最高的一步。

### 第二步
再学会：
- 用 skill 沉淀领域知识和 checklist

这样能减少重复解释。

### 第三步
再尝试：
- team 分工
- reviewer / tester / architect 三角协作

### 第四步
最后在真正需要时再上：
- worktree 隔离
- 多任务并行开发

---

## 9. 真正高频可复用的模板建议

如果你只想马上提高使用效率，建议先准备这些：

### Agent 模板
- reviewer
- tester
- architect

### Skill 模板
- code review checklist
- test workflow checklist
- release checklist
- repo-specific conventions

这样已经能覆盖很多真实工作场景。

---

## 10. 最后的压缩建议

如果只记最短版：

- 小任务：主 agent
- 局部独立问题：subagent
- 高频角色：做成自定义 Agent
- 领域流程/清单：做成 Skill
- 持续分工：team
- 并行改代码：worktree

以及：

> **不要为了用功能而用功能，要让复杂度只在真的需要时出现。**
